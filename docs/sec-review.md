# Security Review: yb Rust Implementation

**Date:** 2026-03-21
**Scope:** `rust/yb-core/` and `rust/yb/` (complete source tree)
**Reviewer:** Claude Sonnet 4.6 (automated static analysis)

---

## Executive Summary

Overall security posture is **strong**. The codebase uses modern authenticated
encryption, performs zero-knowledge private-key operations entirely inside the
YubiKey hardware, and contains zero `unsafe` blocks. The move from the
deprecated Python implementation (AES-256-CBC without authentication, secrets
passed in subprocess arguments) to this Rust implementation eliminated every
critical and high-severity finding from the prior review.

No critical or high-severity findings were identified in the current Rust
implementation. Three low-severity observations and two informational notes
are recorded.

---

## 1. Cryptographic Implementation (`yb-core/src/crypto.rs`)

### Encryption scheme

New blobs are encrypted with **hybrid authenticated encryption**:

| Component | Choice | Notes |
|---|---|---|
| Key agreement | ECDH P-256 (ephemeral-static) | Private key never leaves YubiKey |
| Key derivation | HKDF-SHA-256 | IKM = raw shared-secret x-coordinate; info = `"hybrid-encryption"` |
| AEAD | AES-256-GCM | 96-bit random nonce per message from `OsRng` |

All choices follow current best practice. The nonce birthday bound (2^48
messages before 2^-32 collision probability) is orders of magnitude beyond
any realistic blob count.

The **legacy read path** (AES-256-CBC, version byte `0x04`) is decryption-only;
no CBC ciphertext is ever written by the current implementation.

Peer public-key validation is handled by the underlying `p256` crate, which
rejects invalid curve points. ✓

### GCM authentication failure message

The opaque library error is wrapped and re-raised as:

> "AES-GCM authentication failed — blob may be corrupted or tampered"

No internal cryptographic detail escapes into the error path. ✓

## 2. PIV Session Handling (`yb-core/src/piv/session.rs`, `tlv.rs`)

### Management key mutual authentication

The standard three-pass GENERAL AUTHENTICATE protocol (NIST SP 800-73-4) is
implemented correctly:

1. Card presents an encrypted witness (challenge).
2. Host decrypts the witness, generates a fresh random host challenge, returns
   both.
3. Host verifies the card's response matches the expected encryption of the
   host challenge.

The host challenge is generated byte-by-byte with `rand::random::<u8>()`,
which is backed by `OsRng`. The algorithm identifier (3DES / AES-128 / AES-256)
is dispatched correctly in `tlv.rs`. ✓

### Timing observation (Low — S-01)

`session.rs:229` uses the `!=` operator for the management-key auth comparison:

```rust
if challenge_enc != *challenge_resp { … }
```

Rust's slice equality is not guaranteed to be constant-time, so a sufficiently
precise local timing oracle could in theory distinguish matched from mismatched
bytes. In practice, the round-trip latency over USB PC/SC (milliseconds)
dominates by several orders of magnitude, making exploitation impractical.
Replacing this with `subtle::ConstantTimeEq` would be a trivial one-line
change and would satisfy a defence-in-depth requirement.

**Severity:** Low | **Action:** Optional.

### PIN verification

PINs are padded to 8 bytes with `0xFF` before embedding in the VERIFY command —
the correct YubiKey-specific format. Retry counter (`0x63Cx`) and blocked
(`0x6983`) status codes are handled explicitly. ✓

## 3. Store Format (`yb-core/src/store/mod.rs`, `constants.rs`)

### Bounds safety — `unwrap()` in LE helpers (Low — S-02)

```rust
pub(crate) fn read_u32_le(buf: &[u8], offset: usize) -> u32 {
    u32::from_le_bytes(buf[offset..offset + 4].try_into().unwrap())
}
```

The `unwrap()` panics on slice-length mismatch. Today this is unreachable:
`Object::from_bytes` validates `buf.len() >= OBJECT_MIN_SIZE` (512 bytes) before
calling these helpers, and all defined field offsets are ≤ 23 bytes. The
precondition is, however, implicit and invisible at the call site — a future
refactor could silently introduce a panic.

**Severity:** Low | **Recommendation:** Return `anyhow::Result<u32>` and
propagate the error explicitly, making the invariant visible to the compiler.

### Chunk-chain cycle guard ✓

`chunk_chain()` maintains a `HashSet` of visited indices and terminates on
re-entry, preventing an infinite loop on a corrupted store. Covered by the
`chunk_chain_cycle_terminates` unit test. ✓

### Store sanitization ✓

Duplicate-named heads are resolved by retaining the highest-age entry;
orphaned continuation chunks are zeroed. Both behaviours are tested. ✓

## 4. Orchestrator (`yb-core/src/orchestrator.rs`)

### Blob name validation ✓

`validate_name()` rejects empty names, names longer than 255 bytes, names
containing null bytes, and names containing `/`. UTF-8 validity is enforced
implicitly by Rust's `String` type. ✓

### Capacity check before commit ✓

`store_blob` calculates the required chunk count, verifies sufficient free
slots exist (accounting for any blob being replaced), and only then begins
writing. If the store is full it returns `Ok(false)` — no partial write
can occur. ✓

## 5. CLI & Secret Handling (`rust/yb/src/`)

### PIN exposure via `--pin` / `YB_PIN` (Low — S-03)

The `--pin` flag and `YB_PIN` environment variable allow passing the PIN
non-interactively. Command-line arguments are visible in process listings
(`ps aux`, `/proc/*/cmdline`); environment variables are visible to
child processes and, on some systems, to other users with `ptrace` rights.

These are documented convenience options; the secure default path
(`rpassword` interactive prompt) is unaffected. Users who choose to use
`--pin` or `YB_PIN` in automated workflows should be aware of the
OS-level exposure.

**Severity:** Low | **Action:** Document the caveats in README security section.

### PIN in memory (Informational — I-02)

The PIN is cached in a `RefCell<Option<String>>` for the duration of the
process. Rust does not zero `String` memory on drop. For a short-lived CLI
process executing a single operation this is acceptable. If the PIN is
treated as higher-value material, the `zeroize` crate provides explicit
zeroing on drop.

**Severity:** Informational | **Action:** Consider `zeroize` if raising the
security bar.

### Default credential detection ✓

If the connected YubiKey still has the factory-default PIN (`123456`), PUK,
or management key, the operation is blocked and a warning is printed. The
check can be bypassed with `--allow-defaults` or `YB_SKIP_DEFAULT_CHECK` for
test automation. ✓

### Argument parsing ✓

Clap-based, type-safe. Mutually exclusive output flags (`--stdout` /
`--output` / `--output-dir`) are checked explicitly. Multi-file store
performs a capacity pre-check before any writes (all-or-nothing semantics). ✓

## 6. VirtualPiv / Test Infrastructure

Hardcoded credentials (`PIN=123456`, management key
`010203040506070801020304050607080102030405060708`) appear in
`virtual_piv.rs` and in the YAML fixture files. These are clearly labelled
as disposable test material and are never reachable outside
`#[cfg(test)]` / `#[cfg(feature = "virtual-piv")]` code paths.

**Severity:** Informational (I-01) | **Action:** None required.

## 9. Comparison with Prior Python Implementation

The December 2025 review of the Python implementation identified two critical
issues. Both are resolved in the current Rust implementation:

| Prior finding | Python status | Rust status |
|---|---|---|
| AES-CBC without authentication | Critical — unauthenticated | Resolved — AES-256-GCM |
| Secrets in subprocess args (`--key=…`) | High — visible in `ps` | Resolved — no subprocesses; PC/SC in-process |
| No dependency pinning | High | Resolved — `Cargo.lock` |
| Debug output leaks key material | High | Resolved — no such debug path |
| Command injection via reader name | High | Resolved — no subprocesses |
